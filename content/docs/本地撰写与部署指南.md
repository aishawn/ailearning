# 本地撰写文档 + 部署到网站 — 维护指南

适合「在本地用编辑器写 MD，改完推代码/重新部署」的流程。重点：**目录怎么放、图片怎么管、用什么工具**。

---

## 一、推荐目录结构

```
项目根/
├── content/
│   └── docs/                    # 所有文档 MD/MDX 放这里（和现在一致）
│       ├── 编辑与管理说明.md
│       ├── VLA_世界模型/
│       │   ├── WoG-字节港大.md
│       │   └── ...
│       └── VLA_info/
│           └── ...
├── public/                      # 网站静态资源，部署后会从根路径提供
│   └── docs-images/              # 文档用图（推荐放这里，路径稳定）
│       ├── VLA_世界模型/        # 按主题或目录分子文件夹，避免重名
│       │   ├── wog-arch.png
│       │   └── ...
│       └── VLA_info/
│           └── ...
```

**要点**：

- **正文**：只动 `content/docs/` 下的 `.md` / `.mdx`，和当前 fumadocs 配置一致。
- **图片**：统一放 `public/docs-images/`，按主题或目录建子文件夹（如 `VLA_世界模型`），部署后通过根路径访问，不依赖 content 路径解析，最省心。

---

## 二、图片在 MD 里怎么写（部署后能正常显示）

Next 只会把 `public/` 下的文件当静态资源提供，所以文档里要用 **以 `/` 开头的绝对路径** 指向 `public` 里的文件。

**推荐写法**（和上面结构对应）：

```markdown
![WoG 架构图](/docs-images/VLA_世界模型/wog-arch.png)
```

- 不要用 `../public/...` 或 `./images/...`（content 里的相对路径在构建/部署时不一定能解析）。
- 同一主题的图放在同一子目录，例如 `public/docs-images/VLA_世界模型/`，便于管理和避免重名。

**可选：和 content 同名的子目录**（方便本地对位）：

- 例如有 `content/docs/VLA_世界模型/xxx.md`，就建 `public/docs-images/VLA_世界模型/` 放该主题的图，MD 里用 `/docs-images/VLA_世界模型/fig1.png`。

## 三、本地用什么编辑器（好管理 MD + 图片）

### 1. Typora（最省心，优先推荐）

- **适合**：想专注写、少折腾，又要经常插截图/示意图。
- **用法**：
  - 用 Typora 直接打开 **`content/docs`** 或某个子目录，当“根”来写。
  - 粘贴图片时：**偏好设置 → 图像 → 选择「复制到指定路径」**，路径填：`public/docs-images` 或按主题填子路径（如 `public/docs-images/VLA_世界模型`）；插入时选「相对路径」或「绝对路径」看下面说明。
- **路径建议**：若 Typora 只能插入相对路径（如 `../public/docs-images/VLA_世界模型/fig.png`），部署会挂；所以更稳妥的是：
  - 在 Typora 里先 **插入图片** 选「复制到指定路径」到 `public/docs-images/主题名/`，
  - 然后在 MD 里**改写成绝对路径**：`/docs-images/主题名/文件名.png`。
- **管理**：用左侧文件树在 `content/docs` 里切文章；图片统一在资源管理器里看 `public/docs-images` 即可。

### 2. Obsidian（适合多文档、双链、标签）

- **适合**：文档多、喜欢双链/标签/图谱，想一个库管全部解读。
- **详细配置与操作**（库选哪、图片存哪、路径怎么改）：👉 **[Obsidian 配置与操作](./Obsidian-配置与操作.md)**
- **简要**：库选项目根目录 → 设置「新附件默认位置」为 `public/docs-images` → 在 `content/docs` 下写 MD，粘贴图后把相对路径改成 `/docs-images/...`。

### 3. VS Code（轻量、和代码同环境）

- **适合**：已经用 VS Code 看代码，顺手改 MD。
- **插件建议**：
  - **Markdown All in One**：目录、快捷键、预览。
  - **Paste Image**（或 **Markdown Image**）：粘贴图片时保存到文件并插入路径。
- **设置**：把「粘贴保存路径」设成 `public/docs-images` 或按主题的子目录；插入后若生成的是相对路径，自己改成 `/docs-images/...` 以保证部署可用。
- **管理**：左侧资源管理器切 `content/docs`；图片在 `public/docs-images` 里按目录管理。

**小结**：  
- 想**最少配置、专注写作**：Typora + 图片复制到 `public/docs-images/主题/`，MD 里写绝对路径。  
- 想**多文档/知识库**：Obsidian，同一约定。  
- 想**和代码一起**：VS Code + 粘贴图片插件，同一路径约定。

## 四、本地撰写 → 部署的流程

1. **本地写/改**：在 `content/docs` 下改 MD，图片放到 `public/docs-images/...`，文中用 `/docs-images/...`。
2. **本地预览**（可选）：在项目根跑 `pnpm dev`，打开 `/docs/...` 对应路由看效果。
3. **提交**：`git add content/docs public/docs-images`，`git commit`，`git push`。
4. **部署**：按你现在的部署方式（Vercel / 自建 Node 等）**重新构建并发布**；构建会读最新的 `content/docs` 和 `public/`，无需额外步骤。

只要图片都在 `public/docs-images`、MD 里用绝对路径，部署后就能正常显示。

---

## 五、建议的日常习惯

| 习惯 | 说明 |
|------|------|
| 一篇文档的图放同一子目录 | 如 `public/docs-images/VLA_世界模型/` 下只放该主题的图，文件名简短可读（如 `arch.png`、`result-table.png`）。 |
| 文件名避免中文和空格 | 用英文或拼音，如 `wog-arch.png`，减少部署/缓存问题。 |
| 改完 MD 顺手检查图片路径 | 确认是 `/docs-images/...` 且文件确实在 `public/docs-images/...` 下。 |
| 大图先压缩再放 | 用 TinyPNG 或本地工具压一下再放进 `public/docs-images`，加载更快。 |

---

## 六、和「站内编辑」的关系

- **本地撰写 + 部署**：改的是同一批文件（`content/docs` + `public/docs-images`），部署后站点看到的即本地内容。
- **站内编辑**（Admin → 内容文件 (MD)）：是**在浏览器里**改 `content/docs` 里的 MD，图片会走「上传到站内存储」并插入 URL；不涉及 `public/docs-images`。
- 两种方式**不要混用同一篇文章的图片**：要么这篇都用本地图（`/docs-images/...`），要么都用站内上传的 URL，避免一半图在 public、一半在 CDN 难以维护。

按上面方式维护 MD 和图片，本地写好再推送、重新部署，网站就会和本地一致、图片也能稳定显示。
